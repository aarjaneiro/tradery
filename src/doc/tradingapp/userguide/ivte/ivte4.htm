To create a new session select New from the file menu or click on the new session on the application toolbar. To populate the session with plug-in configurations, select one or more plug-in and/or plug-in configurations of any type and combination desired in the plug-in explorer bar and drag and drop them on the new session’s configuration bar.

 

Session runtime parameters such as the time period for data, number of threads for runnable execution or position sizing can be set in the session parameters bar.

 

Note: currently the only position sizing parameter is the Initial Capital value. More options will be added in future releases.

 

Once the session has been configured, it can be run by clicking on the > button on the session toolbar. If the default datasource is static, the session will run all the runnables once on all the symbols provided by the default symbols source and show the results in the various tab windows. The results are shown in real time except for trades, which are normally shown at the end of the session. If the default data source is dynamic, then the session will not stop after all the symbols have been exhausted, but instead will wait for a notification of new data from the data source, at which moment a new cycle will start. 

 

An automated trading session can be defined by combining a dynamic data source and a signal handler configuration that can connect to a broker.

 

Note: all runnables in a session run simultaneously, in different threads, (and possibly on different CPUs), and they all send their output information (generated trades and trading signals, errors, output) to the same windows or components belonging to the session.

 

A session can be saved using the Save command. The various elements of the session are saved in a file with the default extension ".tap", in XML format.

 

The TradingApp Platform comes with pre-configured data files and symbol sources, which can be selected in the plug-in explorer. There are also several pre-configured session files in the "sessions" directory.

 

Existing session files (.tap) can be opened using one or more options:

  • from the File/Open menu or by clicking on the respective toolbar button

  • from the command line (see command line options below) - one or more sessions

  • by dragging and dropping one or more session files from a Windows file explorer window onto the IVTE main window

  • by double clicking on a session file in a Windows file explorer window - the installation procedure associates .tap files with the IVTE application

 

Note: all configurations of a session are dynamically loaded - i.e. the DLLs they reside in are loaded when the session run starts and released when the session run ends. In case of dynamic data sources, they will stay loaded until the data source signals that there is no more data available and the session ends. Thus, when testing various plug-ins, the corresponding DLLs can be re-build between runs, without having to shut down the GUI application. There is one exception though, in case the same plug-in is used by more than one session, in which case the DLL may stay loaded even if one of the session is stopped.

Command line

Several parameters can be passed to the IVTE on the command line. 

Any number of session file names can be passed on the command line and, if valid, they will be opened at startup. These session file names do not need to be preceded by a parameter name.

It is also possible to specify the plug-in discovery path on the command line using the optional parameter -p or(--pluginpath) followed by the plug-in discovery path. The other plug-in parameters (recursive or not, and extension) are taken from the application settings.

The option -? or --help will bring up a dialog box with information about supported command line parameters.

For both types of arguments, the file name or path must be a quoted string if it contains space characters.

Example:

tradingapp.exe -p c:\plugins \session1.tap "c:\new sessions\session2.tap"

tradingapp.exe --pluginpath c:\plugins

The first example will set the plug-in discovery path to c:\plugins and it will open session1 and session2 sessions. The second example only sets the plug-in discovery path.
